## 1. リサーチャーによる調査結果

### 1-1. 日本の人気アプリに見る共通パターン

| アプリ | 勝ちパターン | 本プロジェクトへの示唆 |
|--------|-------------|----------------------|
| **LINE** | 白基調・シンプル、アイコン+テキスト併用のボトムナビ、「反響がないことが成功」の自然なUI刷新 | ナビゲーションは必ずアイコン+テキスト。派手さより自然さ |
| **PayPay** | トップに最重要情報（残高）を大きく配置、金融横断でUI統一 | コレクション画面の最上部に進捗バーを大きく配置 |
| **メルカリ** | ミニマルUI、余計な表示を省いた「目的達成型」 | 1画面1目的の原則。情報の引き算を徹底 |
| **Yahoo! JAPAN** | 高密度でも一覧性を確保、適度な余白 | カードギャラリーは情報密度と余白のバランスが鍵 |
| **任天堂** | 「UI自体が娯楽」、全年齢が楽しめる、テキストに頼りすぎない | 開封演出はUIの一部として楽しめるものに |
| **ポケポケ** | 紙のカード感＋デジタル演出、12h/1回の無料開封で期待感 | カードの「モノ感」を大切にした質感表現 |

### 1-2. 日本人ユーザーの心理的特性

#### 「安心感」と「丁寧さ」の表現

- **丁寧語**: 説明文・エラー文は「〜です」「〜ます」で敬意を表現
- **視覚的安心感**: 角丸（曲線）オブジェクト、馴染みのあるレイアウト
- **情報の明示**: 「今どこにいて、次に何をすればいいか」を常に提示
- **謙虚なエラー**: ユーザーを責めないトーン（「入力に誤りがあるようです」）

#### 情報密度の好み（欧米との違い）

| 観点 | 日本 | 欧米 |
|------|------|------|
| 情報密度 | 高密度・一覧性重視 | 余白重視・1要素を大きく |
| 配色 | 多色でカテゴリ分け | ミニマルなカラーパレット |
| 認知パターン | 全体を俯瞰して把握 | 1点に集中して深掘り |
| テキスト量 | 説明文を丁寧に記載 | 短く端的に |

#### 失敗への恐怖心

- 日本人ユーザーは「間違えたらどうしよう」という不安が強い
- **確認画面文化**: 重要操作（削除・送信）には確認ダイアログが期待される
- ただし、すべてに確認を挟むと「確認慣れ」で注意力が低下する
- **最適解**: 取消不能な操作にのみ確認を挟み、それ以外はアンドゥ（取消）で対応

#### 「おもてなし」的UXの定義

> ユーザーが困る前にニーズを察知し、細やかな配慮で先回りすること。

具体例:
- 空のコレクションで「まだカードがありません。最初の1枚を受け取ろう！」と優しく導線提示
- エラー時に「何が起きて」「どうすれば解決するか」をセットで伝える
- 操作完了時にさりげないフィードバック（チェックマーク、振動、色の変化）

### 1-3. 日本特有のUIパターン

#### 確認画面

- **使うべき場面**: データ削除、決済、送信など取り消し困難な操作
- **ボタン文言**: 曖昧な「OK」ではなく具体的な「削除する」「送信する」
- **使ってはいけない場面**: 日常的な操作（お気に入り登録、ページ遷移など）

#### ステップ表示

- 複数段階のフローには `1/3 → 2/3 → 3/3` の進捗表示が期待される
- 日本のユーザーは「あとどれくらいか」が見えないと不安になる傾向
- 最適ステップ数: 3〜4（5以上は離脱率上昇）

#### 丁寧語・敬語のトーンガイド

| 場面 | トーン | 例 |
|------|--------|-----|
| 歓迎・初回 | 丁重＋歓迎 | 「ようこそ！カード収集を始めましょう」 |
| 説明・ガイド | 丁寧語 | 「コードを入力してカードを受け取れます」 |
| エラー | 謙虚＋建設的 | 「コードが見つかりませんでした。もう一度ご確認ください」 |
| 成功 | 祝福＋喜び | 「カードを獲得しました！」 |
| ボタン | 簡潔・能動態 | 「受け取る」「始める」「戻る」（敬語不要） |
| 確認 | 明確な問いかけ | 「データを削除しますか？この操作は取り消せません」 |

#### 空状態・エラー状態のケア

- **空状態**: イラスト/アイコン + 親しみやすい文章 + 次のアクションボタン
- **エラー**: 技術コードではなく平易な説明 + 解決策の提示 + 入力欄直下に表示
- これらは「第一印象のチャンス」。放置は最大の機会損失

### 1-4. モバイルファーストの日本的設計基準

| 項目 | 推奨値 | 根拠 |
|------|--------|------|
| ボトムナビ項目数 | 3〜5個 | 選択肢過多の回避 |
| ナビ形式 | アイコン + テキスト | アイコンのみは意味が伝わりにくい |
| 本文フォント | 14〜16px | Google推奨・可読性の最適値 |
| 行間 | 1.5〜1.75倍 | スマホでの可読性 |
| タッチターゲット | 44px以上（推奨48px） | Apple HIG / Material Design |
| ボタン間隔 | 8px以上 | 誤タップ防止 |
| タブ文言 | ひらがな/カタカナ優先 | 視認性・親しみやすさ |

#### タブ命名のルール

- **ひらがな**: 柔らかさ・親しみ（例:「さがす」「おしらせ」）
- **カタカナ**: 外来語・定着語（例:「ホーム」「コレクション」）
- **漢字**: 端的に伝えたい時（例:「設定」「履歴」）
- **原則**: ターゲットがカジュアル層なら、ひらがな・カタカナ寄り

### 1-5. カード収集文化との接点

#### ガチャ演出の心理学

- 「回す前」が最もドーパミンが出る（期待のピーク）
- 変動比率強化スケジュール: いつ当たるかわからない → やめられない
- **段階的演出が最も効果的**: タップ → 光/色変化 → カード出現 → レアリティ確定
- **スキップ可能は必須**: 繰り返し時のストレス回避

#### コレクション達成感の演出

- **進捗バー**: 全体の何%か常に表示
- **図鑑形式**: 未取得カードをシルエットで見せる（集めたい欲の刺激）
- **カテゴリ管理**: シリーズ別・レアリティ別で整理
- **コンプリート報酬**: バッジ・称号などの達成報酬

## 2. デザイナーによる提案

リサーチ結果を踏まえ、本プロジェクトに適用すべき具体的なデザイン原則を提案する。

### 2-1. デザイン原則 5箇条

#### 原則1: 「迷わせない」 — 1画面1目的

- 各画面には明確な1つのゴールがある
- Landing → 「始める」、Claim → 「コードを入れる」、Collection → 「カードを眺める」
- 画面に入った瞬間、最初にタップすべきものが3秒以内にわかること

#### 原則2: 「安心させる」 — 常に現在地と次の一手を示す

- ヘッダーに現在の画面名を必ず表示
- ボトムナビでアクティブ状態を明示（色 + 太さの変化）
- フローには必ずステップインジケータ（オンボーディング 1/3, 2/3, 3/3）
- ボタンの文言は具体的な動詞（「受け取る」「見る」「戻る」）

#### 原則3: 「喜ばせる」 — 受取の瞬間を最高にする

- 開封演出はこのアプリの「顔」
- レアリティに応じた差分演出（Common は控えめ、Legendary は豪華に）
- 演出は最大2秒、タップで即スキップ可能
- 獲得後に「おめでとう」のフィードバック + コレクションへの導線

#### 原則4: 「寄り添う」 — 失敗や空の状態を丁寧にケア

- 空コレクション: アイコン + 「最初の1枚を受け取ろう！」+ CTAボタン
- エラー: 赤背景に具体的な原因 + 解決策（「もう一度確認してください」）
- 使用済みコード: 「このコードは既に使用されています」と明示
- すべてのエラー文は「ユーザーのせいではない」トーン

#### 原則5: 「所有欲を満たす」 — カードが主役のビジュアル

- カード画像は常に最大・最高品質で表示
- レアリティに応じたグロー（光彩）でカードの格を表現
- コレクション進捗バーで「あと○枚」を常に意識させる
- カード詳細画面ではシリーズ情報・取得日でストーリー性を付与

### 2-2. カラーパレット提案

```
Background:    #030712 (深い濃紺/ほぼ黒)
Surface:       #111827 (カード背景)
Border:        #1f2937 (境界線)
Text Primary:  #f9fafb (白)
Text Secondary:#9ca3af (グレー)
Text Tertiary: #6b7280 (薄グレー)
Accent:        #6366f1 (Indigo — 信頼感と高級感)
Success:       #34d399 (Emerald)
Error:         #f87171 (Red)
Warning:       #fbbf24 (Amber)
```

- **ダークテーマ採用理由**: カードの色彩を最大限引き立てるため
- 日本のゲームアプリはダークテーマが主流（ポケポケ, FGO, プロセカ等）
- カードの「光」や「グロー」がダーク背景で映える

### 2-4. コンポーネント設計方針

#### ボトムナビゲーション

```
[コレクション]  [受け取る]  [設定]
  LayoutGrid      Gift     Settings
```

- 3タブ構成（シンプル・迷わない）
- 「受け取る」を中央に配置（最も使用頻度が高い主要アクション）
- アクティブ: Indigo色 + テキスト表示
- 非アクティブ: Gray500

#### ボタン階層

| 種類 | 用途 | スタイル |
|------|------|---------|
| Primary | 主要アクション（受け取る、始める） | Indigo背景・白文字・角丸xl・py-4 |
| Secondary | 副次アクション（別のカードを受け取る） | Gray800背景・Border・白文字 |
| Ghost | 控えめなアクション（スキップ、戻る） | テキストのみ・Gray500 |
| Danger | 破壊的操作（データ削除） | Red背景・白文字 |

#### カードコンポーネント

- サムネイル: アスペクト比 5:7（トレカの標準比率に近い）
- 角丸: 12px（rounded-xl）
- ボーダー: レアリティカラーの2px実線
- グロー: `box-shadow` でレアリティカラーの光彩
- ホバー/タップ: `scale(0.95)` のフィードバック

## 3. クリティカルデザイナーによる批評と改善

### 3-1. デザイナー提案への批評

#### 批評1: 情報密度が低すぎないか

> **問題**: 現在の設計は欧米的なミニマリズムに寄りすぎている。日本人ユーザーは
> 「一覧性」を好む。カードギャラリーが2カラムでは、8枚のカードを見るのに
> スクロールが必要になり、コレクションの全体像が把握しづらい。

**改善案**:
- スマートフォンでは2カラムを維持（カード画像の視認性とのトレードオフ）
- ただし、コレクション画面上部に「ミニマップ」的なシリーズ進捗を追加
- 「一目で全体像がわかる」感覚を進捗バーとシリーズ概要で補完する

#### 批評2: ダークテーマ一辺倒のリスク

> **問題**: ダークテーマは確かにカードを引き立てるが、日本の主要アプリ
> （LINE, PayPay, メルカリ）は白基調。ダーク一辺倒は「なじみのなさ」を
> 生む可能性がある。

**改善案**:
- ダークテーマは「カード体験画面」（開封・詳細・ギャラリー）に限定し、
  カードの色彩を活かす合理的な理由がある場面で使用する
- ただし、本プロジェクトはゲーム/コレクション文化圏のアプリであり、
  ポケポケ・FGO・プロセカ等の先例からダークテーマは許容範囲と判断
- **結論: ダークテーマ維持。ただし、コントラスト比はWCAG AA基準を厳守**

#### 批評3: オンボーディングは本当に必要か

> **問題**: リサーチでは「説明の前にカードに触れさせる」が勝ちパターン。
> 3ステップのオンボーディングモーダルは、ユーザーを「やりたいこと」から
> 遠ざけている。任天堂は「テキストに頼りすぎない」と言っている。

**改善案**:
- オンボーディングモーダルは残すが、「スキップ」を最も目立つ位置に
- 理想は「初回ログイン後、即座にデモカード1枚を自動付与」して体験させる
- モーダルより「ツールチップ型」のコンテキストガイドが日本的おもてなし
- **結論: モーダルは維持（MVP段階）。ただしスキップを強調し、将来的にはコンテキストガイドへ移行を推奨**

#### 批評4: 演出スキップ後の体験が空虚

> **問題**: 演出をスキップしたユーザーの体験が考慮されていない。
> スキップ → 突然カードが表示される → 「あれ、もう終わり？」となる。

**改善案**:
- スキップ時も最低限のフィードバックを提供（0.3秒のフェードイン + バッジ表示）
- スキップは「演出カット」であって「体験カット」ではない
- **結論: スキップ時にも軽量な達成感フィードバックを実装する**

#### 批評5: エラー文言が「説明不足」

> **問題**: 「無効なコードです。もう一度確認してください」では、
> ユーザーは何を確認すればいいかわからない。

**改善案**:
- 「無効なコードです。大文字・ハイフンを含む形式（例: PHOENIX-2024）で入力してください」
- エラー文には常に「具体的な次のアクション」を含める
- **結論: エラー文にフォーマット例を追加**

#### 批評6: 「図鑑形式」の未取得カード表示が欠如

> **問題**: リサーチでは「未取得カードをシルエットで見せる」が
> コレクション欲の重要なドライバー。現在の実装は取得済みカードのみ表示。

**改善案**:
- コレクション画面に「図鑑モード」切替を追加
- 未取得カードはシルエット + 「???」で表示
- シリーズ名とカード番号のみ開示（何を集めればいいか示す）
- **結論: MVP後の優先改善項目として図鑑モードを追加推奨**

### 3-2. 追加で必要な日本的配慮

#### 配慮1: フォーム入力のストレス軽減

- コード入力欄: 自動大文字変換（実装済み）
- 入力中にリアルタイムバリデーション表示は避ける（日本人は入力中の指摘を嫌う）
- 送信ボタンタップ後にまとめてバリデーション

#### 配慮2: ローディング状態の丁寧さ

- 読み込み中はスケルトンUIまたはシマーエフェクトを使用
- 「読み込み中...」ではなく、具体的に「カードを準備しています」等
- スピナーは最低0.3秒表示（瞬間消滅は不安を与える）

#### 配慮3: 触覚フィードバック

- レジェンダリーカード開封時の振動（`navigator.vibrate`）は好印象
- ただし、すべてのタップに振動を入れるのは過剰
- **使うべき場面**: カード獲得成功、レアカード出現のみ

#### 配慮4: データ削除の多段階確認

- 「データの全削除」は日本のユーザーにとって非常に怖い操作
- 現在の実装（1回の確認ダイアログ）は最低限OK
- 理想: 削除対象の具体的な内容を列挙（「カード○枚、アカウント情報」）

## 4. 3者合意: 本プロジェクトへの適用ガイドライン

### 4-1. 最終合意した設計原則

| # | 原則 | 詳細 | 優先度 |
|---|------|------|--------|
| 1 | **1画面1目的** | 各画面の最優先アクションを1つに絞る | 必須 |
| 2 | **現在地の明示** | ヘッダー + ボトムナビで「今どこ・次どこ」を常に表示 | 必須 |
| 3 | **開封体験の最大化** | レアリティ差分演出、スキップ可能、スキップ時も軽量フィードバック | 必須 |
| 4 | **丁寧なエラーケア** | 具体的原因 + フォーマット例 + 次のアクション | 必須 |
| 5 | **空状態のおもてなし** | アイコン + 説明文 + CTAボタンの3点セット | 必須 |
| 6 | **進捗の常時可視化** | コレクション進捗バーを常に表示 | 必須 |
| 7 | **丁寧語の適切な使い分け** | 説明文は丁寧語、ボタンは簡潔な動詞 | 必須 |
| 8 | **確認は重要操作のみ** | データ削除・ログアウトのみ。日常操作には確認不要 | 必須 |
| 9 | **図鑑モード（未取得表示）** | シルエット + ??? で集めたい欲を刺激 | 推奨（MVP後） |
| 10 | **コンテキストガイド** | モーダルからツールチップ型ガイドへの移行 | 推奨（MVP後） |

### 4-2. トーン＆ボイス最終定義

#### アプリ全体のトーン

- **親しみやすいが、馴れ馴れしくない**
- **丁寧だが、堅苦しくない**
- **楽しいが、騒がしくない**

#### 文体ルール

1. 説明文は「です/ます」調（丁寧語）
2. ボタンは動詞の終止形（「受け取る」「始める」「削除する」）
3. エラーは「〜ようです」「〜ください」（ユーザーを責めない）
4. 成功は「〜しました！」（感嘆符で喜びを表現、ただし1つまで）
5. 空状態は「〜しましょう！」（励まし・提案のトーン）

### 4-4. 勝ちパターン / アンチパターン チェックリスト

#### 実装時に確認すべき勝ちパターン

- [ ] ボトムナビはアイコン+テキスト併用になっているか
- [ ] 各画面のヘッダーに画面名が表示されているか
- [ ] フォームエラーは入力欄の直下に表示されているか
- [ ] エラー文に「原因 + 次のアクション」が含まれているか
- [ ] 空状態に「アイコン + 説明 + CTAボタン」があるか
- [ ] 重要操作（削除等）に確認ダイアログがあるか
- [ ] ボタン文言は具体的な動詞か（「OK」「はい」は不可）
- [ ] タッチターゲットは44px以上あるか
- [ ] 開封演出はスキップ可能か
- [ ] コレクション進捗が常に表示されているか
- [ ] 丁寧語は適切に使い分けられているか（ボタンには不要）
- [ ] `prefers-reduced-motion` に対応しているか

#### 絶対にやってはいけないアンチパターン

- [ ] すべての操作に確認ダイアログを出す
- [ ] アイコンのみのナビゲーション（テキストなし）
- [ ] エラーを技術コード（Error 404等）のみで表示する
- [ ] 空の画面を何もない白/黒のままにする
- [ ] 演出がスキップできない
- [ ] 高圧的・命令的なエラーメッセージ
- [ ] 40px未満のタップ領域
- [ ] 「OK / Cancel」の曖昧なボタン
- [ ] ユーザーを責めるトーン（「正しく入力してください」→ NG）
- [ ] 現在地がわからないナビゲーション
